## Session 1 — Root Cause Analysis: Why the Classifier Wasn't Running (June 11, 2026)

---

## Root Cause Analysis — Why the Classifier Wasn't Running

### Context

The project downloads an Edge Impulse person-classifier model for the ESP32-S3 (GOOUUU cam board with 8MB OPI PSRAM). The sketch compiles and uploads without errors, the camera initializes fine, and frames are captured correctly. But every call to `run_classifier()` returned error **-3** (`EI_IMPULSE_TFLITE_ERROR`), meaning TFLite inference failed completely.

---

### Step 1 — First symptom: error -3 with no explanation

The sketch already had a `printEiError()` function that mapped -3 to `"TFLite or inference engine error"`, but gave no detail. We could see the board booting, camera working, and even `get_data` callbacks firing — but `run_classifier()` silently failed every time.

---

### Step 2 — Enabling hidden TFLite error strings

The Edge Impulse library has this in `micro_log.h`:

```cpp
#ifndef EON_COMPILER_RUN
#define TF_LITE_STRIP_ERROR_STRINGS   // ← silences ALL TFLite internal errors
#endif
```

Because `EON_COMPILER_RUN` is not defined (this is a standard TFLite Micro build), `MicroPrintf(...)` becomes a no-op — every internal TFLite error message is stripped out at compile time. This is why the only output was "AllocateTensors() failed" from the high-level wrapper, with no detail below it.

**Fix:** Comment out `TF_LITE_STRIP_ERROR_STRINGS` in `micro_log.h` to enable verbose TFLite logging.

---

### Step 3 — Finding the exact failure message

With error strings enabled, the board printed:

```
Failed to resize buffer. Requested: 258688, available 189916, missing: 68772
AllocateTensors() failed
```

This meant the TFLite Micro arena (memory workspace used to lay out all tensors and scratch buffers) was too small. The model needed **~259KB** inside the arena at peak, but only **~190KB** was available.

---

### Step 4 — Why the arena was too small

The Edge Impulse library defined the arena size in:
```
src/tflite-model/tflite_learn_1013165_3.h
```
as:
```cpp
const size_t tflite_learn_1013165_3_arena_size = 256403;
```

This value is generated by Edge Impulse's profiler — but it appears to have been underestimated for this specific chip/SDK version combination. The actual peak usage during `AllocateTensors()` was ~325–470KB.

Additionally, the default `ei_calloc()` in the library uses `MALLOC_CAP_DEFAULT`, which can allocate from internal RAM (~298KB free at boot) — but TFLite Micro's internal resizing can fail if the allocation pattern doesn't satisfy alignment or contiguity requirements.

---

### The Fixes (both required together)

#### Fix 1 — Force arena allocation into PSRAM

Added to `men_ai_model.ino` (overrides the weak library symbols):

```cpp
void *ei_malloc(size_t size) {
    return heap_caps_malloc(size, MALLOC_CAP_SPIRAM | MALLOC_CAP_8BIT);
}

void *ei_calloc(size_t nitems, size_t size) {
    return heap_caps_calloc(nitems, size, MALLOC_CAP_SPIRAM | MALLOC_CAP_8BIT);
}

void ei_free(void *ptr) {
    free(ptr);
}
```

The board has **8MB PSRAM** free — this guarantees the 700KB arena fits easily.

#### Fix 2 — Increase the arena size

In `~/Arduino/libraries/esp32_person_classifier_inferencing/src/tflite-model/tflite_learn_1013165_3.h`:

```cpp
// Before:
const size_t tflite_learn_1013165_3_arena_size = 256403;

// After:
const size_t tflite_learn_1013165_3_arena_size = 700000;
```

The `--clean` flag was needed when recompiling to force the cached `.o` files to be rebuilt with the new value.

---

### Result after fixes

```

```

The classifier runs correctly at ~147ms per frame (~7 fps equivalent) using the ESP32-S3's NN accelerator.

## Next steps / To continue

### Run the Python listener
The Python script reads serial output (predictions + images) from the ESP32:
```bash
cd men_detection_model
python3 men_ai_model.py
```
Make sure `/dev/ttyUSB0` is available and you have permissions:
```bash
sudo usermod -aG dialout $USER   # then log out/in once
```
